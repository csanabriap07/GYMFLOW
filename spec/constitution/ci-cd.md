# CI/CD y contenedores

> Sin HU/RN/RF de origen en `docs/` — es infraestructura del entregable (requisito de "pipeline con Jenkins o GitHub Actions"), no una regla de negocio. Documentado aquí por seguir el mismo criterio que el resto de `spec/`: nada se decide en silencio.

## Decisiones confirmadas

- **GitHub Actions**, no Jenkins. Razón: sin servidor propio que mantener, se dispara solo con cada push/PR, minutes gratis para este tamaño de repo. Jenkins tendría sentido con un servidor propio ya corriendo, que no es el caso.
- **Alcance del pipeline (por push/PR):** correr tests de backend (pytest) + build de frontend + levantar todo el stack en contenedores como smoke test. **No** despliega a ningún servidor automáticamente — eso queda fuera de este pipeline por ahora.
- **Backend:** imagen basada en `python:3.12-slim`, dependencias con Pipenv (`pipenv install --deploy`, reproducible desde `Pipfile.lock`). El contenedor corre `alembic upgrade head` automáticamente antes de arrancar `uvicorn` (`docker-entrypoint.sh`), para no depender de un paso manual de migración en cada `docker compose up`.
- **Frontend:** build multi-stage — Node compila con Vite, nginx sirve el resultado estático. Se eligió nginx en vez de `vite preview` porque preview no está pensado para producción (no tiene las optimizaciones de un servidor estático real).
- **Comunicación frontend↔backend en el stack de compose:** nginx hace de proxy (`/api/` → `backend:8000/`, `/ws/` → WebSocket del kiosko de `012`), para no depender de CORS dentro del stack "productivo". En desarrollo local (`npm run dev`, sin Docker), el proxy lo maneja Vite (`vite.config.ts`, a configurar cuando exista el código).
- **Postgres:** `postgres:16-alpine`, con volumen nombrado para persistir datos entre reinicios de `docker compose up`.
- **`GET /health`:** vive como criterio de aceptación de `001-checkin-membresia-activa` (el equipo decidió agruparlo ahí en vez de dejarlo como endpoint suelto sin spec). Sin autenticación, responde `200 OK` si el proceso está arriba — no valida base de datos ni reglas de negocio. Lo usan el `healthcheck` de `docker-compose.yml` y el smoke test de CI.

## Duda abierta (a confirmar, no invento la respuesta)
- **`Pipfile`/`Pipfile.lock` y `package.json`/`package-lock.json` ya existen, generados de verdad** (no escritos a mano): `pipenv install` con las dependencias ya decididas en `tech-stack.md` (FastAPI, SQLAlchemy, Alembic, psycopg2-binary, python-jose, passlib, pytest, httpx) y `npm create vite -- --template react-ts` + Tailwind **v3** (fijado a propósito, no v4, para que coincida con la config de `tailwind.config.js` que ya usa `design-system.md`) + React Query + Axios, con **React 18** fijado explícitamente porque `tech-stack.md` lo pide así (el scaffold trae v19 por defecto). `tailwind.config.js` ya tiene los tokens de `design-system.md` cargados.
- **Lo que falta para que `docker compose up` levante algo funcional de verdad:** el scaffold trae la base (dependencias + Tailwind + un placeholder de `App.tsx`), pero **no hay código de aplicación real todavía** — falta `backend/main.py`, la configuración de Alembic (`alembic.ini` + `env.py`), y los módulos (`auth/`, `members/`, `checkin/`, etc.). Eso es la implementación real de las features, no infraestructura — sigue el flujo normal de `spec/README.md` (spec → plan → tasks → código), feature por feature, no se scaffolda todo de una vez aquí.
- **Despliegue real (más allá del smoke test): deliberadamente pospuesto, no es una ambigüedad pendiente.** El equipo decidió priorizar que el stack quede bien automatizado en local (Docker + CI validándolo en cada push) antes de elegir dónde correrlo en la nube — así, cuando se decida, es agregar un job al `ci.yml` (o un `deploy.yml` aparte) sobre una base ya probada, no rehacer nada. Cuando llegue el momento, retomar aquí: dónde se despliega (VPS propio, Railway/Render, algo de la universidad) y cómo se conecta ese job con las credenciales necesarias.
- **Secretos en GitHub Actions:** `JWT_SECRET` real de producción (si aplica) debe ir en **GitHub Secrets**, no en el repo ni en `.env.example`. El `.env.example` solo tiene valores de desarrollo/CI, nunca los reales.
